论文笔记：《Deep Learning based Vulnerability Detection:Are We There Yet?》

Deep Learning based Vulnerability Detection: Are We There Yet?

CCF A类期刊；IEEE Transactions on Software Engineering (Early Access)；2021年

Abstract

最先进的基于dl的技术在真实世界的漏洞预测场景的效果下降了50%以上。原因是现有的基于dl的漏洞预测方法在训练数据和模型选择方面存在问题，通常无法了解与漏洞的实际原因相关的特性。

基于以上原因，本文采用更有原则的方法来收集数据和设计模型，并基于脆弱性预测的现实设置，从而得出更好的解决方案，并为未来基于dl的漏洞预测研究绘制了路线图

I. Introduction

A.现有DL模型存在的问题

  • 模型本身的不足。最流行的模型是基于标记的，它将代码视为标记的序列，而不考虑在漏洞预测中扮演重要角色的语义依赖。即使使用基于图的模型，它也不会关注于增加脆弱和非脆弱类别之间的类分离。因此，在真实世界中，他们具有低精度和低召回率的问题。
  • 学习无关特性。现有的最先进的模型本质上是在获取与漏洞无关的特征，这些特征可能是训练数据集的产物。
  • 数据重复。大多数现有方法的训练和测试数据包含重复数据(高达68%);因此，人为夸大了报告的结果。
  • 数据不平衡。现有的方法并不能缓解类不平衡问题。现实漏洞分布作为非漏洞代码的频率要比漏洞代码的频率高得多。

B.本文提出的方法

REVEAL——更有原则的数据收集和模型构建方法

1) 数据收集

讨论了如何整合真实世界漏洞预测数据，包括静态和进化(即bug修复)性质的漏洞

2) 模型构建

展示了表示学习可以用于传统的DL方法之上，以增加脆弱样本和非脆弱样本之间的类分离。

II.Background and Challenges

最流行的DLVP方法包括三个步骤:数据收集、模型构建和评估。

  • 首先，收集训练数据，根据设计目标和资源约束选择合适的模型。
  • 根据所选模型的格式对训练数据进行预处理。然后对模型进行训练，使损失函数最小。
  • 经过训练的模型将用于现实世界。为了评估模型的有效性，在不可见的测试实例上评估了模型的性能。

A.DLVP理论

基于DL的漏洞预测器从一个训练数据(Dtrain)集中学习漏洞代码模式，其中代码元素被标记为漏洞或非漏洞。给定一个代码元素(x)和相应的脆弱/非脆弱标签(y)，模型的目标是学习关于模型参数(θ)的概率p(y|x)最大化的特征。

P(y|x)表征漏洞元素x有漏洞的概率

通常，这种转换和概率计算是通过模型中的一个前馈层和一个softmax层来实现的。

对于漏洞预测等二元分类任务，通过最小化交叉熵损失来学习最优模型参数。

B.现有的数据集

数据集的类型可分为如下：

  • 合成数据集： SATE IV Juliet、SARD
  • 半合成数据集：Draper、SARD、NVD
  • 真实数据集：Devign（包含来自四个开源项目过去的漏洞和其修复）

越靠近左下角合成度越高，反之真实度越高

局限性

  1. 由简单模式组成的合成数据集上训练的模型将仅限于检测那些在现实生活中很少出现的简单模式，即只能学习到简单的漏洞模式，对于复杂的漏洞模式难以检测
  2. 数据集不能反映真实世界的漏洞代码分布
  3. 数据集一般只含有漏洞代码及其修复，不含有其上下文，所以基于这些数据集的模型也无法学习到漏洞代码和其上下文的关联。使得模型无法区分漏洞函数和其上下文。

C.现有的建模技术

基于标记的模型

这些模型中，代码被认为是标记的序列

  • 简单的基于符号的模型
    • Li等人提出了一种基于双向长短期记忆(BSLTM)的模型
    • Russell等人提出了一种基于卷积神经网络(CNN)和Radom森林的模型

符号序列长度是影响性能的一个重要因素，因为模型很难对长序列进行推理

  • 对代码切片的模型
    • VulDeePecker和SySeVR对代码进行切片

切片动机：并不是代码中的每一行对漏洞预测都同等重要。因此，不考虑整个代码，只考虑从代码中“感兴趣的点”提取的片段(如API调用、数组索引、指针使用等)进行漏洞预测，其余部分不考虑。

基于图的模型

这些模型将代码视为图形，并合并了不同的语法和语义依赖关系。可以使用不同类型的语法图(抽象语法树)和语义图(控制流图、数据流图、程序依赖图、Def-Use链图等)进行漏洞预测

  1. Yamaguchi等人提出的Devign杠杆代码属性图(CPG)，建立他们基于图的漏洞预测模型。CPG是通过在代码的抽象语法树(AST)中增加不同的依赖边(如控制流、数据流、化解等)来构建的**。**

局限性

  1. 基于标记的模型假设标记之间是线性依赖的，因此标记之间只存在词法上的依赖，而失去了语义上的依赖
  2. 基于图的模型可以考虑数据依赖边，但是资源开销更大，在资源受限的环境表现不好
  3. 尽管经过训练的模型学会了区分漏洞和良性代码样本，但训练范式并没有明确地专注于增加漏洞和良性示例之间的分离。因此，稍有变化，分类就会变得脆弱。
  4. 另一个问题是漏洞代码和良性代码之间的数据不平衡，真实世界数据集中漏洞示例与良性示例的比例非常低。当一个模型在这样不平衡的数据集上训练时，模型往往会被良性的例子所误导。

D.评估过程

局限性

所有现有的方法都使用它们自己的评估数据集来报告它们的性能，但它并没有对模型在现实世界中的适用性给出一个全面的概述。能从这种数据集内评估中学到的是，现有的方法如何适合他们自己的数据集。

尽管有一些关于在现实世界的项目中发现漏洞的这种模型的有限的案例研究，这些案例研究并没有阐明假阳性和假阴性。假阳性和假阴性的数量与开发人员在漏洞预测上的工作直接相关，太多的假阳性和假阴性会阻碍开发人员使用模型。

III.REVEAL设计

A.数据收集

通过跟踪两个开源项目(Linux Debian Kernel和Chromium (Chrome的开源项目))过去的漏洞，构建了一个更健壮、更全面的真实世界数据集REVEAL。

过程：

  1. 首先用公开的补丁收集已经修复的问题。对于Chromium，我们删除了它的bug库Bugzilla2。对于Linux Debian内核，我们从Debian安全跟踪器3收集了问题。
  2. 对于每个补丁，提取了补丁中发生变化的C/ c++源文件和头文件对应的漏洞和修复版本(即旧版本和新版本)。将所有更改函数的前版本(即补丁之前的版本)标注为“vulnerable”，将所有更改函数的固定版本(即补丁之后的版本)标注为“clean ”。此外，补丁中没有涉及的其他功能(即那些保持不变的功能)都被标注为’ clean ’

B.模型构建

分为两个阶段，即特征提取和训练。

  • 在特征提取阶段，将现实世界的代码转换为图形嵌入。
  • 在训练阶段，根据提取的特征训练一个表示学习模型，以学习最理想地将脆弱的例子与非脆弱的例子区分开来的表示。

1) 特征提取

该阶段的目标是在保持语义和语法信息的同时，将代码转换为一个紧凑的、长度一致的特征向量。

  1. 生成代码属性图（CPG）
    1. CPG提供了由AST和程序依赖图(或PDG)之外的控制流和数据流图的元素组成的代码的组合和简洁表示。
  2. 结点特征提取
  • 结点由结点类型（算数表达式、函数调用等）和原始代码片段组成。
  • 用独热向量编码结点类型，用word2vec对原始代码片段完成词嵌入将两者连接组成结点的联合向量

  1. GGNN嵌入

2) 训练

a.重采样

原始数据集存在的问题：漏洞类和无漏洞类数量的不平衡

使用“合成少数过采样技术”(简称SMOTE)来处理漏洞类和无漏洞类数量的不平衡，通过改变数据中不同类的频率来操作。

具体来说SMOTE对多数类进行子采样(即随机删除一些示例)，同时对少数类进行超采样(通过创建合成示例)，直到所有类具有相同的频率。在脆弱性预测的情况下，少数类通常是脆弱实例。

  • 在超采样过程中，SMOTE选择一个脆弱例子，找出k个最近的脆弱邻居。然后，它通过在自己和最近的随机邻居之间插入，构建一个少数民族的合成成员。
  • 在欠采样时，SMOTE随机地从训练集中移除非脆弱样本。这个过程会不断重复，直到脆弱和非脆弱示例之间达到平衡。

b.表示学习

原始数据集存在的问题：特征提取后漏洞和无漏洞代码样本的图嵌入在特征空间中趋于高度重叠

使用多层感知器(MLP)将原始不可分空间中的特征投射到一个潜在空间中，在漏洞样本和无漏洞样本之间提供更好的可分离性。以解决漏洞和无漏洞代码样本的图嵌入在特征空间中趋于高度重叠。

MLP设计

损失函数

为了使隐藏空间中漏洞实例和非漏洞实例的分离最大化，采用了三重态损失作为损失函数。三重态损失由三个单独的损失函数组成:

  • (a)交叉熵损失( LCEL_{CE}LCE​);

    • 交叉熵损耗随着预测概率偏离实际标签而增加

  • (b)投影损失(LpL_pLp​);

    • 用于量化隐藏表示如何很好地分离漏洞和非漏洞的例子，如果隐藏空间中所有的漏洞例子彼此靠近，而同时又远离所有的非漏洞例子，即来自同一类别的例子彼此非常接近(即相似)，而来自不同类别的例子彼此很远，则认为隐藏表示是有用的。

  • ©正则化损失(LregL_{reg}Lreg​ )。

    • 用于限制隐藏表示(h(xg))的大小，h(xg)的任意增加会阻止模型收敛，所以使用正则化损失(Lreg)来惩罚在量级上较大的潜在表示(h(xg))
